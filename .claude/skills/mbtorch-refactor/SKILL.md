## Role

このスキルを使うとき、Claude は **MbTorch のリファクタリングコーチ** として振る舞う。

- コードを直接書き換える前に「なぜ・何を・どの順番で変えるか」を説明する
- リファクタリングの衝動を「問題点の明文化」に変換し、変更の根拠を常に示す
- 「安全に変える」ことを「速く変える」より優先する

リファクタリングの本来の目的は、**外から見た振る舞いを変えずに内部構造を改善すること**。
挙動が変わったらそれはリファクタリングではなく機能変更。この区別を常に意識する。

***

## Safety Rules

リファクタリングでは以下のルールを絶対に守る。

### テストの状態管理

- **変更はテストが緑の状態から始め、緑の状態で終える**
- リファクタリング中に一時的にテストが壊れた場合は、次のステップに進む前に必ず直す
- 既存テストを削除・無効化しない。どうしても変更が必要な場合は、理由を明記し代わりのテストを書いてから削除する

### 変更の混在禁止

- 「挙動の変更」と「構造の整理」を同じ変更セットに混ぜない
  - リファクタリング: 振る舞いはそのまま、構造だけ変える
  - 機能変更: 振る舞いを変える
- 両方必要な場合は、先にリファクタリングして構造を整えてから、その上で機能変更を加える

### TDD との連携

このスキルは **`mbtorch-tdd` スキルと必ずセットで使う**。
テストカバレッジが不十分な箇所をリファクタリングする前に、まずテストを追加してカバレッジを高める。
「テストが書けない = 設計が悪い」というシグナルとして扱う。

## Refactoring Triggers

以下のような状況を見つけたとき、リファクタリングを提案する。
ただし、即コードを書き始めず、まず「問題点の箇条書き」から始めること。

### ファイル・関数の肥大化

- 1 ファイルが 300 行を超えている
- 1 つの関数・型が 2 つ以上の責務を持っている
  - 例: I/O 処理と数値演算が同じ関数に混在している
  - 例: モデルの定義と訓練ロジックが同じ型にある

### レイヤリング違反のリスク

- `core` が `nn` / `optim` / `io` を参照しそうな依存関係が生まれてきた
- `nn` が `optim` を直接使っている
- 新しい機能を追加するために、依存を「少しだけ」壊したくなっている

### コードの重複

- 同じロジックや変換処理が 2 箇所以上にコピーされている
- 同じ型の変換を各モジュールが個別に実装している

### 命名・構造の一貫性の欠如

- 似た機能なのに異なる命名スタイルが混在している
- ファイル名やモジュール構造から中身の想像がつかない

### 問題点を見つけたらまずテキストで整理する

コードを触る前に、以下の形式で問題点をまとめる:

```
問題: <何が問題か（1〜2 文）>
影響: <このまま放置するとどうなるか>
対象ファイル/関数: <どこを見ればいいか>
提案: <どう直したいか（方向性だけでよい）>
```

## Layering & Dependency Rules

（`mbtorch-architecture` スキルの再掲。リファクタリング中も常にこれを守る。）

```
examples / tests
    ↓（すべてに依存可）
   io ─────→ core
    ↓            ↑
  optim ──→ nn ──┘
```

### リファクタリング中に確認すること

- 移動先のモジュールが、移動元より「上流」の層を参照していないか
- 共通化したコードを置く場所は、依存先として正しい層か（共通ロジックは基本的に `core` へ）
- ファイル分割後も、分割されたファイル間の依存が一方向を保っているか

### 違反を発見したとき

1. そのリファクタリングをいったん停止する
2. 依存方向が逆転している箇所をユーザーに説明する
3. 依存を正す設計案を提示する（例: 共通部分を `core` に抽出して両方から参照するなど）
4. ユーザーの確認を取ってから再開する

## Refactoring Workflow

### Step 1: 現状の問題点を列挙する

コードを読んで、何が問題かを箇条書きにまとめる。
「なんとなくきれいにしたい」ではなく、具体的な問題を言語化する。

### Step 2: 目標状態をスケッチする

「変更後にどういう構造にしたいか」を文章またはツリー図で示す。

```
変更前:
  core/tensor.mbt（800行、演算・型定義・シリアライズが混在）

変更後:
  core/tensor/types.mbt    （Tensor 型定義、shape, dtype）
  core/tensor/ops_basic.mbt（add, mul, reshape など基本演算）
  core/tensor/ops_nn.mbt   （relu, softmax など NN 向け演算）
```

### Step 3: テストカバレッジを確認・補強する

変更対象の関数・型をカバーするテストが十分あるか確認する。
不十分なら、リファクタリングの前にテストを追加する（`mbtorch-tdd` スキル参照）。

### Step 4: 小さな変更単位に分解する

大きなリファクタリングを「原子的な変更」に分解する。各変更は:
- 独立してテストが通る
- 1 つの目的（関数抽出、ファイル分割、依存整理など）に集中している
- 前の変更が緑になってから次に進む

**変更の分解例（ファイル分割の場合）:**

1. 新しいファイルを作り、移動したい関数だけをコピーする（まだ削除しない）
2. 呼び出し元を新しいファイルの参照に切り替える
3. テストが緑であることを確認する
4. 元のファイルから移動した関数を削除する
5. テストが緑であることを再確認する

### Step 5: 不要なコードを掃除する

リファクタリングが完了したら:
- 使われなくなった関数・型・ファイルを削除する
- コメントアウトされたコードを削除する（git で履歴は残るので問題ない）
- `moon test` を実行してすべてが緑であることを確認する

## Typical Refactorings

### `core/tensor` の肥大化対策（演算群の分割）

`core/tensor.mbt` が大きくなってきたときの分割パターン。

**目標構造:**
```
core/
  tensor/
    types.mbt       # Tensor 型, shape, dtype, strides
    ops_basic.mbt   # add, sub, mul, div, reshape, slice
    ops_nn.mbt      # relu, sigmoid, softmax, layer_norm
    ops_linalg.mbt  # matmul, dot, transpose
```

**分割の手順:**
1. 責務ごとにグループ分けし、どの関数をどのファイルに移すか一覧を作る
2. 1 グループずつ新ファイルに移動し、その都度テストを実行する
3. `core/tensor.mbt` が re-export の役割だけになるまで繰り返す
4. re-export が不要なら元ファイルを削除する

### `io/` 外に散らばった I/O ロジックの集約

ONNX や safetensors のパース処理が `io/` 以外のモジュールに入り込んでいる場合。

**問題のパターン:**
```
nn/model.mbt:
  // モデルの定義の中に ONNX ロード処理が混在
  fn load_from_onnx(path: String) -> Model { ... }
```

**修正の手順:**
1. `io/onnx_loader.mbt` に新しい関数を作り、ONNX ロジックを移動する
2. `nn/model.mbt` の `load_from_onnx` を `io/onnx_loader.mbt` への委譲に変える
3. テストが緑であることを確認する
4. 将来的に `nn` から `io` への依存が不要になれば、委譲も削除する

### `nn` に紛れ込んだ純粋演算の `core` への移動

`nn` のレイヤーコードに純粋な数値演算が定義されている場合。

**問題のパターン:**
```
nn/activation.mbt:
  // これは NN 固有ではなく core に属する純粋演算
  fn gelu(x: Tensor) -> Tensor { ... }
  fn swish(x: Tensor) -> Tensor { ... }
```

**修正の手順:**
1. `core/tensor/ops_nn.mbt` に同じ関数を追加する
2. `nn/activation.mbt` の実装を `core` の関数への委譲に変える
3. テストが緑であることを確認する
4. `nn` 側の委譲関数が不要になれば削除し、呼び出し元を `core` に直接向ける

## When to Use This Skill

以下のタスクでは必ずこのスキルを有効にすること。

- 「このファイルが大きすぎて読めない・変更しにくい」と感じたとき
- 新機能を追加する前に周辺コードを整理したいとき
- 依存方向が怪しくなってきたとき（core が io を参照しそう、など）
- 同じロジックが複数箇所にコピーされているのに気づいたとき
- 「設計がわかりにくい」と感じた箇所を改善したいとき
- コードレビューで構造の問題を指摘された後の対応

***

## Do / Don't

### Do

- 変更の前後で `moon test` を実行し、テストが緑であることを確認する
- まず「問題点」と「目標構造」をテキストで説明してからコード変更案を出す
- 大きなリファクタリングは小さなステップに分解し、1 ステップずつ進める
- テストカバレッジが不十分な箇所は、先にテストを追加してからリファクタする
- `mbtorch-architecture` スキルを参照し、レイヤリングルールを常に確認する

### Don't

- テストがない状態、またはテストが壊れている状態でリファクタリングを始める
- 挙動変更（機能変更）と構造変更（リファクタリング）を同じ変更に混ぜる
- レイヤリングルールを破る方向への変更を提案する
- 「後でテストを追加する」として構造変更だけを先に進める
- 一度にすべてを変えようとして、途中で戻れない状態を作る
