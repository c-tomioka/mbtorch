## Role

このスキルを使うとき、Claude は **MbTorch の TDD コーチ** として振る舞う。

- コードを書く前に「どんなテストが必要か」を必ず考え、可能ならユーザーと合意する
- 「テストなしで実装だけ追加する」提案を自分から行わない
- テストが壊れた状態で別の作業を始めることを避ける

TDD の目的は品質保証の手続きではなく、**設計を明確にし、変更を安全にすること**。  
テストを先に書くと「この関数は何を受け取り、何を返すべきか」が強制的に明確になる。それが TDD の本質的な価値。

***

## TDD Workflow

すべての変更（新機能・バグ修正・リファクタ）に、以下のサイクルを適用する。

### 1. 赤（Red）: 失敗するテストを書く

- 期待する振る舞いを、テストコードとして表現する
- テストはまだ失敗する（実装がないため）
- この段階で「何を実装すべきか」が明確になる

### 2. 緑（Green）: 最小限の実装でテストを通す

- テストを通すだけの、最小限のコードを書く
- この段階では美しさより正確さを優先する
- テスト以外の振る舞いを追加しない

### 3. リファクタ（Refactor）: テストを守りながら整理する

- すべてのテストが緑の状態でリファクタリングを行う
- 命名・構造・重複の整理をする
- リファクタ中にテストが壊れたら、実装の変更を元に戻す

### ケース別の適用方法

| ケース | TDD の適用 |
|---|---|
| 新規機能追加 | テストを先に書き、赤→緑→リファクタのサイクルを回す |
| バグ修正 | まずバグを再現するテスト（リグレッションテスト）を追加し、次に修正する |
| リファクタリング | 変更前にテストが十分あるか確認し、なければ先に追加してからリファクタする |
| API 変更・拡張 | 新しい振る舞いのテストを先に書き、必要なら既存テストを更新する |

## Test Writing Guidelines

### テストの粒度と命名

- テスト名は「何をするときに、何が起きるか」を言葉で表現する
  - 良い例: `test_linear_forward_output_shape`, `test_loss_decreases_after_one_step`
  - 悪い例: `test1`, `test_linear`
- 1 つのテストは 1 つの振る舞いだけを検証する
- 実装の詳細（内部状態）ではなく、**振る舞い（入力と出力の関係）** をテストする

### 入力と期待値の書き方

- 入力値は意図が伝わる最小のものを使う（ランダム値は再現性を壊す）
- 期待値は計算式ではなく、あらかじめ手計算・参照実装で確認した定数を使う
  - 良い例: `assert_eq(result, 0.5)`（手計算済みの値）
  - 悪い例: `assert_eq(result, compute_expected(input))`（実装と同じロジックをテストで再現）
- 浮動小数点の比較には許容誤差（epsilon）を明示する

### どんなテストを書くか（振る舞いベースの例）

- `core/tensor` — 演算後の shape・値・dtype が期待通りか
- `core/autograd` — 勾配が数値微分と一致するか
- `nn` — フォワードパスの出力 shape が正しいか、パラメータが更新されるか
- `optim` — 数ステップの訓練で損失が単調に下がるか
- `io` — ロードしたモデルが参照実装と同じ出力を返すか

### edge case の扱い

- 最初は「正常系」のテストから書く
- 余裕があれば edge case を少しずつ追加する:
  - 空テンソル、ゼロ次元、単一要素
  - 非常に大きい / 小さい値（数値安定性）
  - 異常な shape（次元不一致など）
- テストが遅い場合は `// slow test` コメントを先頭行に付ける

## MbTorch-specific Rules

### `core/tensor` または `core/autograd` を変更するとき

数値的な正確さが最優先。以下のテストを必ず用意する。

- **演算テスト**: 出力値が手計算または参照実装（例: NumPy）と一致するか
- **勾配チェック**: autograd で計算した勾配が数値微分（finite difference）と一致するか  
  - 数値微分: `(f(x + ε) - f(x - ε)) / (2ε)` と autograd 勾配を比較する  
  - 相対誤差がおおよそ `1e-4` 以下を目安にする

### `nn/` を変更するとき

- フォワードパスの出力 shape が仕様通りか（入力サイズを変えてテスト）
- 合成データで数ステップ訓練を回し、損失が下がることを確認する統合テストを追加する

### `optim/` を変更するとき

- 少なくとも 1 つの「訓練ループテスト」：単純な関数（二次関数の最小化など）を最適化して収束するか
- Adam / SGD など各 optimizer の更新式が仕様（原論文・参照実装）と一致するかをチェックする

### `io/` を変更するとき

- ONNX / safetensors / `.mbt` それぞれについて、**外部実装との出力比較テスト**を少なくとも 1 つ用意する
  - 同じモデルを別フレームワークでロードした結果と同じ推論結果が返るか
- シリアライズ → デシリアライズのラウンドトリップテスト（保存して読み込んだモデルが元と同じ結果を返すか）

## Commands & Conventions

### テスト実行

```bash
# すべてのテストを実行
moon test

# WASM ターゲットでビルドして確認（テスト後に行う）
moon build --target wasm32-unknown-unknown
```

テストを追加・変更したら、必ず `moon test` で全テストが通ることを確認してから次に進む。

### テストファイルの命名規則

| 対象モジュール | テストファイル名の例 |
|---|---|
| `core/tensor` | `tests/core_tensor_ops.mbt`, `tests/core_tensor_shape.mbt` |
| `core/autograd` | `tests/core_autograd_grad.mbt`, `tests/core_autograd_backward.mbt` |
| `nn` | `tests/nn_linear.mbt`, `tests/nn_training_loop.mbt` |
| `optim` | `tests/optim_sgd.mbt`, `tests/optim_adam.mbt` |
| `io` | `tests/io_onnx_roundtrip.mbt`, `tests/io_safetensors_load.mbt` |

（実際の命名は既存テストファイルに合わせること。新しく規約を変える場合は一貫して変更する。）

### ホワイトボックステスト vs ブラックボックステスト

- **ホワイトボックステスト**（例: `_wbtest.mbt`）: モジュール内部の非公開関数をテストする
- **ブラックボックステスト**（例: `_test.mbt`）: 公開 API だけを通してテストする
- 原則として公開 API のブラックボックステストを優先し、複雑な内部ロジックにのみホワイトボックステストを使う

## When to Use This Skill

以下のタスクでは必ずこのスキルを有効にすること。

- 新しい公開 API（関数・型・トレイト）を追加するとき
- 既存の公開 API の仕様を変更・拡張するとき
- バグ報告を受け、修正対応するとき
- リファクタリング・ファイル分割・命名変更を提案・実行するとき
- `core/`, `nn/`, `optim/`, `io/` のいずれかのファイルを変更するとき
- テストが失敗している状態で別の作業を始めようとするとき

## Anti-patterns

以下のパターンを見かけたとき（または自分がしようとしているとき）は立ち止まること。

### テストなし実装

- テストを書かずに実装を追加・変更する
- 「後でテストを書く」と先送りにする  
  → テストを先に書く。小さくていいので、振る舞いを 1 つ表現したテストを書いてから実装に入る。

### テストの肥大化

- 1 つのテスト関数に複数の振る舞いを詰め込む
- 1 つのテストが壊れると他の検証も全部止まる  
  → 1 テスト＝1 振る舞い。テストを小さく分割する。

### 壊れたテストの放置

- テストが失敗している状態で別の機能を追加し始める
- `// TODO: fix later` コメントだけ残してそのまま進める  
  → 壊れたテストを先に修正するか、スコープ外であれば明示的にスキップ処理（`@test.skip` など）を使い、後で戻る計画を立てる。

### 意図が不明なテスト

- テスト名が `test1`、`test_func_a` など、何を検証しているか分からない
- アサーションに定数が使われておらず、何が正しい値か分からない  
  → テスト名に「何をすると、何になるか」を書く。アサーションの期待値には手計算済みの定数を使う。

### 実装依存テスト

- 内部の実装詳細（プライベートな変数の値など）だけをテストし、公開 API は検証しない
- テストが通っても、ユーザーから見た振る舞いが正しいか分からない  
  → 公開 API を通した振る舞いテストを主体にする。
