## Context

### MbTorch とは

MbTorch は **MoonBit** で書かれた軽量 AI/ML フレームワークで、以下を目標とする。

- WebAssembly / ブラウザ / エッジデバイスをファーストターゲットとした小バイナリ・高速実行
- MoonBit ネイティブで小〜中規模ニューラルネットワークを定義・訓練できる API
- **ONNX** および **safetensors** による学習済みモデルのインポート
- MbTorch ネイティブ形式（`.mbt`、設計中）による効率的なモデル保存・読み込み
- ブラウザ上での推論と軽量ファインチューニング

### 非目標（現時点）

- 大規模分散学習
- ONNX オペレータの完全カバレッジ
- GPU 専用バックエンド

## Repository Layout

```text
mbtorch/
  core/
    tensor/      # テンソル型・前向き数値演算（shape, dtype, スライス等）
    autograd/    # autograd エンジン・勾配計算ロジック
  nn/            # レイヤー（Linear, Conv, Activation 等）・高レベルモデル API
  optim/         # Optimizer（SGD, Adam 等）
  io/            # モデル I/O（ONNX / safetensors / .mbt インポート・エクスポート）
  examples/      # ブラウザ / エッジ / CLI デモ・エンドツーエンドスクリプト
  tests/         # 単体テスト・統合テスト
  .claude/       # CLAUDE.md・skills（本ファイルを含む）
```

### 各ディレクトリの責務

| ディレクトリ | 責務 | 依存してよい先 |
|---|---|---|
| `core/tensor` | テンソル型と前向き数値演算 | 外部ライブラリのみ |
| `core/autograd` | autograd グラフ・勾配定義 | `core/tensor` のみ |
| `nn` | レイヤー・モデル定義・高レベル API | `core` のみ |
| `optim` | パラメータ更新アルゴリズム | `core`, `nn` |
| `io` | モデルシリアライズ・デシリアライズ | `core`, `nn` |
| `examples` | デモ・サンプルコード | すべて |
| `tests` | テスト | すべて（ただしライブラリロジックを再実装しない） |

## Layering Rules

### 許可される依存関係

```text
examples / tests
    ↓（すべてに依存可）
   io ─────→ core
    ↓            ↑
  optim ──→ nn ──┘
```

### NG パターン（絶対に行わない）

- `core` が `nn` / `optim` / `io` / `examples` を import する
- `nn` が `optim` / `io` を import する
- `core/tensor` が `core/autograd` を import する（逆方向は OK）  
  - 理由: テンソルは「勾配なし」も含む純粋な数値レイヤとして保ち、  
    推論専用ビルドや将来のバックエンド分離をしやすくするため
- `io` が `optim` を import する

### ルール違反を発見したとき

1. **すぐに実装を止める**
2. 依存方向が逆転していることをユーザーに説明する
3. 依存を正しい方向に直すリファクタ案を提示する
4. ユーザーの確認を取ってから進める

## Workflow

コードを変更・追加する前に、必ず以下の手順を踏む。

### 1. 変更対象モジュールを特定する

- 変更したい機能はどの層に属するか？（tensor / autograd / nn / optim / io）
- その層のディレクトリにある既存ファイルをすべて読む

### 2. 既存コードを確認する

- 同じ型・関数・トレイトがすでに存在しないか検索する（検索機能を活用）
- 似た処理があれば、まず再利用・拡張を検討する
- 新規追加が必要な場合も、既存の命名規則・型設計に合わせる

### 3. テストから書く（TDD）

- 新しい公開 API には必ずテストを先に書く
- バグ修正には必ずリグレッションテストを追加する
- テストは `tests/` 配下か、モジュールに近い場所に置く

### 4. 最小限の変更を実装する

- テストを通すための最小の実装にとどめる
- 関係のないリファクタや機能追加は別 PR に分ける

### 5. ドキュメントを更新する

- 公開 API を変更・追加した場合は README / `examples/` を更新する
- 使い方スニペットをコメントか example として残す

## Do / Don't

### Do

- コードを書く前に関連モジュールを読み、既存設計を把握する
- レイヤリングルールに沿った依存関係になっているか毎回確認する
- 型・関数・トレイトの再利用を最初に検討する
- テストを先に書いてから実装する
- 公開 API の変更時はマイグレーションパスを提示する
- `core` は純粋な計算ロジックのみを持つよう維持する
- 命名を既存コードと統一する（`Tensor`, `Linear`, `Model.from_onnx` 等）

### Don't

- `core` から `nn` / `optim` / `io` / `examples` への依存を追加する
- テストなしで新しい公開 API を追加する
- 既存の公開 API の振る舞いを黙って変更する
- 機能変更と無関係なリファクタを同じ変更に混ぜる
- 一時的な `examples/` のロジックをライブラリ本体にコピーする
- グローバルな可変状態を導入する
- shape / dtype / エラー条件を暗黙にする（重要なパスでは明示する）
