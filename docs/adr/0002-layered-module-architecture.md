# 0002: レイヤード・モジュールアーキテクチャの採用

* Status: accepted
* Date: 2026-03-07
* Deciders: @c-tomioka, Claude Code

## Context and Problem Statement

MbTorch は tensor 演算、自動微分、ニューラルネットワーク層、最適化、モデル I/O など複数の責務を持つ。これらをどのようなモジュール構成で整理し、依存関係をどう管理するかを決める必要があった。

モジュール間の依存が循環したり、下位モジュールが上位モジュールに依存するとテストや保守が困難になるため、明確なレイヤリングルールが求められた。

## Decision Drivers

- モジュール間の責務を明確に分離したい
- 循環依存を防ぎ、テストを容易にしたい
- 各モジュールを独立して開発・テストできるようにしたい
- 将来的な拡張（新しい層、新しいオプティマイザ等）を安全に追加できるようにしたい
- MoonBit のパッケージシステム (`moon.pkg`) と整合する構成にしたい

## Considered Options

- レイヤード・アーキテクチャ（core → nn → optim → io の一方向依存）
- フラット構成（全モジュールを同一レイヤーに配置）
- プラグイン方式（core のみ固定、他は動的に差し替え可能）

## Decision Outcome

Chosen option: "レイヤード・アーキテクチャ", because:

- `core` (tensor + autograd) を最下層として独立性を保証できる
- `nn` は `core` のみに依存し、層の追加が安全に行える
- `optim` は `core` と `nn` に依存し、パラメータ更新ロジックを集約できる
- `io` は `core` と `nn` に依存するが、逆方向の依存は禁止される
- MoonBit の `moon.pkg` による明示的なインポート宣言と自然に整合する

## Pros and Cons of the Options

### Option 1: レイヤード・アーキテクチャ

#### Pros
- 依存方向が明確で、循環依存が構造的に防止される
- 各レイヤーを独立してテストできる
- 新しいモジュール追加時の影響範囲が限定される
- CLAUDE.md でルールを明文化し、AI アシスタントにも遵守させやすい

#### Cons
- レイヤー間で共有したい汎用ユーティリティの配置に悩む場合がある
- 厳格すぎると、レイヤーをまたぐ便利な API を提供しづらくなる可能性

### Option 2: フラット構成

#### Pros
- モジュール間の参照が自由で、開発初期は素早く動ける

#### Cons
- 循環依存が容易に発生する (assumption)
- 責務の境界が曖昧になりやすい
- プロジェクト成長時に保守が困難になる

### Option 3: プラグイン方式

#### Pros
- 高い柔軟性と拡張性

#### Cons
- 初期の設計・実装コストが高い (assumption)
- MoonBit の現状のモジュールシステムでは動的ロードが困難 (assumption)
- 小〜中規模プロジェクトには過剰な複雑さ
